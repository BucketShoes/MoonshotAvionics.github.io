PACKET TYPES (0x80-0xFF)
These define the over-the-air and over-WebSocket message framing. The type byte values are chosen to be unlikely to appear by coincidence in unrelated traffic.


```
0xAF  Telemetry packet - a compact minimal value for every packet, followed by one data page which cycles through different pages each time
      uint8   type (0xAF)
      uint8   device ID (rocket)
      uint16  GPS lat fractional (0.00002° steps within 
              nearest degree cell, 65535=invalid fix)
      uint16  GPS lon fractional (0.00002° steps, >50000 for error: GPS_FRAC_NO_FIX=65533, GPS_FRAC_INITIALISING=65534, GPS_FRAC_NOT_POWERED=65535)
      int16   Fusion altitude, meters MSL (-32768 = invalid)
      uint16  state flags
              [3:0]   flight phase (0=idle, 1=pad_ready, 2=boost,
                      3=coast, 4=apogee, 5=drogue, 
                      6=main_deploy, 7=descent, 8=landed,
                      15=error)
              [4]    armed
              [5]    pyro/chute ch1 fired/released
              [6]    pyro/chute ch2 fired/released
              [7]    pyro/chute ch3 fired/released
              [8]    low battery
              [9] baro_ok, [10] accel_ok, [11] arm_ready
              [12:15] reserved
      --- 10 bytes fixed, then optionally: ---
      uint8   data page type ID
      (N bytes data page payload, size determined by type ID)
      --- total 10 bytes (no page) to ~21 bytes (10-byte page) ---

0x9A  Command packet - see command listing for details of each command.
      uint8   type (0x9A)
      uint8   target device ID
      uint8   command ID (see command list)
      uint32  nonce (monotonic, never reused, stored in NVS)
      (0-N bytes command params, depends on command ID)
      uint8[10]  HMAC-SHA256 truncated to 10 bytes
                 (computed over all previous bytes)
      --- total 17 bytes (no params) to 25 bytes (8 bytes params) ---

0xE2  Backhaul envelope (relay → base, on backhaul channel, transmitted by relays occasionally, on backhaul channel)
      uint8   type (0xE2)
      --- relay current state (8 bytes) ---
      uint8   relay device ID
      uint8   relay battery (encoded mV)
      uint8   relay primary channel index
      uint8   relay backhaul channel index
      uint32  relay current time, ms since boot
      --- latest recieved packet (from log format) ---
      (N bytes, as per log format - this will be the latest log that is a telemetry packet, excluding length but including its SNR and timestamp)
      --- total: 9 + N bytes ---

0xCA  Log download chunk (used during lora log download sessions only, typically with different radio settings, streamed to WebSocket by base station when recieved)
      uint8   type (0xCA)
      uint8   source device ID
      uint32  start record index (absolute, monotonic)
      uint8   record count in this chunk
      (N bytes: concatenated flash records, each with their 
       own length/type/timestamp/payload as stored in flash)
      --- total: 7 + N bytes, N limited by radio packet size ---

0xBB  Long range packet - This packet is typically transmitted with special radio settings to enable maximum range.
As such, it has an extremely limited payload. When sent over special radio settings, the packet type is inferred and only the 4 byte core is sent. In all other cases (normal radio, logs, parsing, etc, the packet will have normal headers added, not just the core. i.e. the packet type 0xBB and length, etc are all inferred and included before logging
The intention of this packet is to get close. More packets gets you closer statistically due to dithering. Once close, you can get into radio range for normal telemerty, the normal telem will give the precision to actually locate the rocket. Limited packets gives useful information in even any one packet; but more information from multiple due to dithering of values.
Typically this is sent in WIN_LR time slots. Please implement WIN_LR slot type. Because this uses implicit headers, the reciever must know the expected settings: SF:12, BW: 125, Coding Rate: LI4/5, LDRO: ON, Preamble: 5, Header disabled(implicit), Length: 3 bytes. CRC:Off.
This packet is also valid over normal radio settings (with the packet type 0xBB and the device id included)
    uint8 type (0xBB) (skipped in on-air format in special radio modes. present in all other formats, and valid if handled in normal radio mode with this type present)
    uint8 device id of the transmitter
    ===== core payload (the following is included in all transports)
    payload bits:
    [0..10] latitude fractional decimal digits within current cell, dithered. 0-1999 valid. this represents steps approx 75meters.  value >=2000 = error
    [11..21] longitude. e.g. for 151.1234567, ignore 151 (cell assumed from launch location); use 0.1234567*2000=246.91356 so 91.356% chance of rounding up (to 247) else round down (to 246)
    [22] low battery - 100% true at 3400mv or below. 0% true at 3700mv or above
    [23] reserved




```

DATA PAGE TYPES: (these pages are included at the end of a telemetry packet; but can also be logged directly, include details from specific sensors)
Payloads are size sensitive, so these are all tightly packed and these dont represent the underlying units. Currently the largest page is 11 bytes, and bigger than this limits our allowable modulations vs airtime.
Thrust curve is special being so long, but it has special radio handling to be so big.

      int32   lat, degrees × 1e7
      int32   lon, degrees × 1e7

0x0C  Radio health (7 bytes)
      uint16  delayed TX count (times TX was postponed due to
              channel busy)
      uint16  invalid RX count (packets received that failed
              validation — wrong type, wrong device ID, bad CRC,
              etc)
      int8    background RSSI, dB (average only measured when not transmitting and when no command incoming - to estimate channel noise outside of packet times)
      uint8   sync and slot flags
              [0]     radioSynced (1 = tight, 0 = wide. if currently using tight listen windows for power saving, because we have recently heard a valid command during a listen. if no commands heard, window becomes wider. false if using wide listen windows (pre-sync or no recent command heard)
              [7:1]   current slot sequence index (seqIdx % SLOT_SEQUENCE_LEN)

0x0D  Timestamp (8 bytes)
      uint64   UTC time, ms since 1/1/1970


0x0E  Thrust curve — X-axis acceleration history (variable length)
      uint16  durationMs  — window duration in milliseconds
      int16   minAccel8   — min sample, 8mg units (actual mg = val × 8)
      int16   maxAccel8   — max sample, 8mg units
      uint8[] samples     — N bytes; count inferred from record/packet length
                            each = ((val_mg − min8×8) × 255) / ((max8 − min8) × 8)

      Sample count varies by transport:

LOG FORMAT:
```
uint8   length (payload bytes, excluding log header, 0-254. 0xFF = erased/end marker)
int8    SNR (dBm, from radio reception. 0x7F = not from radio / locally generated)
uint32  timestamp (ms since boot on the device which generated/received it)
--- then length bytes of payload (starting with type byte, <=0x7F is raw data page, >=0x80 is recieved packet) ---

```


FLASH INDEX ENTRY (4 bytes):
```
uint32  data partition virtual offset of the first record in this 256-record chunk. The N-th index entry points to the address of the (N*256)th data record.
        (this virtual pointer will exceed the size of the ring buffer after it has looped. Chunks requested more than ringsize bytes before the current log position are refused as overwritten, although indexes may still exist pointing to nonexisting data). 0xFFFFFFFF = unused/erased/end of log marker.
```

In general, values clamp at their min/max unless otherwise stated.


BLE GATT SERVICES AND CHARACTERISTICS:

Two independent BLE GATT services exist — one on the base station and one on the rocket.
The browser page can connect to both simultaneously (Chrome/Edge Web Bluetooth).

--- Base Station BLE Service ---
Service UUID:  4d4f4f4e-5348-4f54-4253-000000000000
+
  Char 0001 (NOTIFY) — Telemetry stream
    Same binary framing as WebSocket messages from the base station.
    Each notification is one WebSocket-equivalent message:
      float32  SNR, dB (little-endian)
      float32  RSSI, dBm (little-endian)
      int32    flash record number (little-endian)
      (N bytes) packet payload — either a 0xAF telemetry packet or 0xCA log chunk
    SNR/RSSI sentinel: 0x7F (127) = no radio path (locally generated/BLE-originated).
    Max notification size: limited by negotiated ATT MTU.

  Char 0002 (WRITE WITH RESPONSE) — Command
    Write: raw binary command packet (same 0x9A format as over-the-air, see above).
    Read after write: single byte result code (0x00 = accepted, nonzero = error).

  Char 0003 (READ) — Status JSON
    Equivalent to GET /api/status over HTTP.
    Returns JSON: {"uptimeMs":N,"records":N,"oldest":N,"baseBattMv":N,"nonce":N}
    uptimeMs: base station milliseconds since boot (for chart x-axis sync).
    nonce: highest command nonce seen (client should use nonce+1 for next command).

  Char 0004 (WRITE | NOTIFY) — Log fetch
    Write: [startRec u32 LE][count u16 LE] — request a range of flash records.
    Notifications: stream of log records in same format as GET /api/logs HTTP response:
      uint32  record number (little-endian)
      uint8   payload length
      int8    SNR (dB, 0x7F = local/no-radio)
      uint32  timestamp ms since boot (little-endian)
      (N bytes) payload (starts with type byte)
    Empty notification (0 bytes) = end of requested range.

  Char 0006 (WRITE | WRITE WITHOUT RESPONSE | NOTIFY) — OTA firmware chunk
    Write (chunk): [offset u32 LE][data: 1–508 bytes]
      Max frame: 512 bytes (4-byte offset + 508 bytes data = Web Bluetooth write limit). Not authenticated individually.
      Chunks must be sent in strictly sequential order (offset == bytesWritten). Gap → status 0x03.
    Notifications (device → client):
      Only sent on errors and periodic progress (no per-chunk ACK — use WRITE_NR for throughput):
      [status byte]  error codes 0x01–0x07 (see command listing for full table)
      [0xA0][bytesWritten u32 LE]  progress report (5 bytes), sent every 1200 chunks (~600 KB)
    OTA session gating: CMD_OTA_BEGIN (0x50) must be sent first via char 0002;
      CMD_OTA_FINALIZE (0x51) and CMD_OTA_CONFIRM (0x52) also go via char 0002.
      See command listing for full OTA state machine and status byte table.

--- Rocket BLE Connection Behaviour ---

  MTU: negotiate 517, but limit notification payloads to BLE_MAX_PDU (502 bytes).
    Rationale: 502 bytes fits in two LL segments; 517 spills into a third.

  PHY: advertise on legacy 1M only (NimBLE dual advertising not fully supported).
    On connect: attempt to negotiate 2M PHY. Default PHY for historical log fetch is 2M.
    Coded PHY S8 is available via char 0004 command 0x03 (lower throughput, longer range — intended for in-flight realtime telemetry).

  Advertising interval: ~1 second (to conserve battery).

  Connection multiplicity: one client at a time. A new connection causes the previous
    connection to be disconnected, buffers and counters cleared, and all page fresh-flags
    set. Advertising continues while connected.

  Realtime telemetry (char 0001):
    On connect, subscription begins at 10 Hz for pages 1–13 (all implemented data pages).
    A "fresh" flag per page type is set when new sensor data arrives; cleared when the page
    is sent. Non-fresh pages are skipped in reliable transports (BLE = reliable).
    If a captured set of pages does not fit in one PDU, the overflow is buffered (max 4 KB)
    and drained one PDU per loop iteration. No new capture begins until the buffer is empty.
    All records in one capture share the same uint64 capture timestamp.

  Historical log fetch (char 0005):
    One PDU per main loop iteration, filled with as many whole records as fit (max 502 bytes).
    Non-blocking. Allowed while armed (impact expected minimal; may be reviewed after flight testing).

  Transport trust: BLE is untrusted. Unauthenticated requests (char 0004 ConnSet, char 0005 log
    fetch) are available to any connected client. Authenticated 0x9A commands (char 0002) still
    require valid HMAC. If BLE is found to affect flight determinism, the entire stack will be
    stopped when armed (not currently implemented).
