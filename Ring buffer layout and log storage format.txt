logs have an index partition and a data partition - due to variable sized log records, the indexes are used to find a numbered record without scanning through the entire data partition - at most 255 records.

the logs go to a ring buffer in the 6mb partition with a virtual index and a virtual address. e.g. if its a 6MB ring buffer, we might be 40MB into it - that means loop around to 40%6=2MB address; but also, disallow reads requesting before (40-6=34MB position) - since those have already been overwritten. 
never allow direct position access, only record number access, which goes via indexes to find position.
One 32kb partition for indexes - 4 bytes per index entry, to point to the offset of the data of the first record in each chunk of 256 records.
One 6MB partition for data records. To find record N, you look at index[N>>8], check if too far back, otherwise look at data of that mod RingSize, then scan through (N & 0xFF) records forward.
On boot, you find the correct place to continue writing by scanning for the first index pointing to offset 0xFFFFFFFF (erased, unwritten), and the one before that is the last chunk, then scan from that starting point until you find a record with type and  length=0xFF, meaning its not written yet. This is the position and counter where the next log record should go. (keep counter and position in ram).

If a record wont fit in the current 4kb sector, put it at the beginning of the next sector (and in the background, begin wiping the following page - always keep the next page cleared/erased ready to go).
The indexes/counters don't point to the actual location, but to the virtual location, which can be beyond the end of the ring - if anything points to something more than 6MB (RingSize) before the current erased page, then that's from a previous loop and its already gone.
"Chunks" refer to the first of a block of 256 records (by count); "Sectors" refer to 4kb (by size). Indexes entries are 4 byte uint32's, which their value is a byte position within the data partition as a long continuous 4gb span. If the current record ever reaches 4gb byte position in data, or reaches 2^32 index position in counter, then logging is not allowed until you send the log erase command (this is months or years of data, and would have used about 1% of =NOR wear lifetime for a 6MB ring - more wear for a smaller ring - normally we expect to just be running ahead of whatever logs we had in the ring from last time, but the offset should warn in JS at 3 milllion).
data ring byte position and index ring counter position loop independently, so there may be indexes which refer to already erased data records - hence why the continuous span and the current position check vs ringsize, which implies whats been erased.
